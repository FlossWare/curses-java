# ADR 0004: Evaluate Rust/UniFFI for Shared Rendering Logic

**Status:** Rejected  
**Date:** 2026-07-18  
**Deciders:** Project Maintainer  
**Tags:** architecture, rust, ffi, code-sharing, spike

## Context

The curses-java (Java 21, FFM calling ncurses) and curses-themes (Python, calling ncurses via stdlib) projects share approximately 125 lines of rendering logic for 3D effects (shadow rendering, highlight/lowlight edge coloring, border drawing). Both projects already share theme *definitions* through a JSON schema (`themes/schema.json`) established in issue #264.

The question is whether Rust, compiled to a shared library (`.so`/`.dylib`) and called via UniFFI-generated bindings, is a viable approach to eliminate the remaining rendering code duplication across Java and Python.

### Maintenance Cost Estimate

The duplicated rendering code has been stable since initial implementation. Estimated ongoing maintenance cost of the duplication: 1-2 hours per year for keeping both implementations in sync when rendering logic changes. Estimated one-time cost of introducing Rust/UniFFI: 2-4 weeks for initial setup (Cargo crate, UniFFI bindings, CI for cross-platform builds, Maven and setuptools integration), plus ongoing per-release effort for cross-platform binary compilation and distribution.

## Evaluation

### 1. Build Complexity

A Rust/UniFFI pipeline would require three build systems cooperating:

**Cargo (Rust):**
- A `cdylib` crate containing the shared rendering functions
- UniFFI proc-macros or UDL file defining the cross-language interface
- `cargo build --release` to produce `libcurses_render.so`

**Maven (Java):**
- `exec-maven-plugin` to invoke `cargo build --release`
- `exec-maven-plugin` to invoke `uniffi-bindgen-java generate --library target/release/libcurses_render.so --out-dir generated-sources/`
- `build-helper-maven-plugin` to add generated Java sources to compilation
- Native `.so` bundled into JAR resources
- JVM flag `--enable-native-access=ALL-UNNAMED` required at runtime

**setuptools (Python):**
- `uniffi-bindgen` to generate Python ctypes bindings
- Native `.so` bundled into wheel or installed alongside package
- Build scripts to invoke Cargo before packaging

**Total toolchain:** Cargo + rustc + uniffi-bindgen + uniffi-bindgen-java + Maven + setuptools. Contributors would need Rust toolchain installed even for pure Java or Python changes that touch rendering. CI would need Rust on all target platforms.

### 2. Binary Size

A minimal Rust `cdylib` has a baseline of approximately **2.2 MB** on Linux due to static linking of the Rust standard library. With aggressive optimization:

```toml
[profile.release]
opt-level = "z"       # Optimize for size
lto = true            # Full link-time optimization
codegen-units = 1     # Better optimization
panic = "abort"       # Remove unwinding machinery
strip = true          # Strip symbols
```

The library could be reduced to approximately **1.5-2.0 MB**, but it cannot go significantly below that floor without abandoning `std` (which would require `unsafe` code and lose access to most of the Rust ecosystem).

For comparison, the rendering logic itself is approximately 125 lines across both projects. The Rust `.so` would contain roughly 50-80 lines of actual Rust code wrapped in 2 MB of runtime overhead.

### 3. FFM Integration (Java)

The `uniffi-bindgen-java` project (by IronCore Labs, version 0.4.2) generates Java bindings using the FFM API (Project Panama) -- the same API this project already uses in `NcursesBridge.java`. The generated code would look similar to the existing FFM patterns:

```java
// Generated by uniffi-bindgen-java (conceptual example)
public class CursesRender {
    private static final MethodHandle drawShadow = linker.downcallHandle(
        lookup.find("curses_render_draw_shadow").orElseThrow(),
        FunctionDescriptor.of(JAVA_INT, JAVA_INT, JAVA_INT, JAVA_INT, JAVA_INT)
    );

    public static void drawShadow(int x, int y, int width, int height) {
        try {
            drawShadow.invokeExact(x, y, width, height);
        } catch (Throwable e) {
            throw new RuntimeException("FFI call failed", e);
        }
    }
}
```

This approach is technically feasible. However, `uniffi-bindgen-java` carries notable risks:
- **Stability:** Self-described as "unstable" with no 1.0 release; API may change between versions
- **Maintenance:** Maintained by a single organization (IronCore Labs) primarily for their own product (ironcore-alloy); not a broadly community-maintained project
- **Rust version coupling:** MSRV is Rust 1.87.0, which pins the project to a recent Rust toolchain
- **Scope:** Currently targets IronCore Labs' use cases; edge cases in our rendering API may not be well-supported

### 4. Python ctypes

UniFFI generates Python bindings using `ctypes` automatically:

```python
# Generated by uniffi-bindgen (conceptual example)
import ctypes

_lib = ctypes.cdll.LoadLibrary("libcurses_render.so")

def draw_shadow(x: int, y: int, width: int, height: int) -> None:
    _lib.curses_render_draw_shadow(x, y, width, height)
```

This is straightforward and well-supported. UniFFI's Python bindings are the most mature of its non-mobile targets.

### 5. Developer Experience

**Rust's theoretical benefits:** Rust offers memory safety guarantees and performance advantages over Java and Python. However, these benefits are not material for this use case: the rendering logic performs no heap allocation (it passes integer coordinates to ncurses), has no concurrency (rendering is single-threaded within a lock), and is not performance-critical (terminal refresh rates are 30-60 Hz at most). The safety and performance characteristics of the existing Java/Python code are already sufficient.

**Rust learning curve:** Rust's ownership model, lifetimes, and borrow checker represent a significant learning curve. For a project whose core logic is in Java, requiring contributors to learn Rust for 125 lines of rendering code is a disproportionate burden.

**Debugging cross-language:** When a bug appears in rendering, the developer must determine whether it is in:
- The Rust source code
- The UniFFI binding generation
- The FFM/ctypes calling convention
- The Java or Python consumer code

Each layer adds diagnostic difficulty. Stack traces across FFI boundaries are opaque. The existing approach (native Java and Python implementations) produces standard stack traces and is debuggable with standard IDE tooling.

**Platform portability:** The Rust `.so` must be compiled for each target platform (Linux x86_64, Linux aarch64, macOS x86_64, macOS aarch64). The current approach requires zero native compilation since both Java FFM and Python ctypes call the system-installed ncurses directly.

### 6. Comparison with JSON-Only Approach

The JSON theme sharing approach (issue #264) already eliminates the most impactful duplication -- the theme definitions that would otherwise drift between implementations. The remaining ~125 lines of rendering logic consist of:

- **Shadow rendering** (~40 lines): Drawing an L-shaped shadow at an offset using shadow color
- **Highlight/lowlight edge coloring** (~50 lines): Applying asymmetric colors to border edges based on RenderingStyle (RAISED/SUNKEN/FLAT)
- **3D border drawing** (~35 lines): Drawing borders with highlight/lowlight color application

This logic is:
- **Simple:** Straightforward loops over coordinates with color attribute calls
- **Stable:** The rendering algorithm has not changed significantly since initial implementation
- **Platform-coupled:** It calls ncurses functions (mvaddch, attron, attroff) that are already wrapped differently by each language (Java FFM vs Python curses module)

To share via Rust, the Rust code would also need to call ncurses -- meaning the Rust `.so` would wrap ncurses, then Java/Python would call the Rust wrapper. This adds an indirection layer (Java FFM -> Rust FFI -> ncurses) compared to the current direct path (Java FFM -> ncurses).

## Decision

**Rejected.** We will not pursue Rust/UniFFI for shared rendering logic.

The cost-benefit analysis is decisively against this approach:

| Factor | Rust/UniFFI | Current (JSON + native) |
|--------|------------|------------------------|
| Shared code | ~125 lines of rendering | ~125 lines duplicated |
| Build complexity | 3 build systems + Rust toolchain | Standard Maven + setuptools |
| Binary overhead | ~2 MB .so for ~80 lines of Rust | Zero additional binaries |
| Contributor barrier | Must know Rust + Java + Python | Java or Python only |
| Debugging | Cross-language FFI stack traces | Standard IDE debugging |
| Platform builds | Must cross-compile .so per platform | None (uses system ncurses) |
| Dependency risk | uniffi-bindgen-java is pre-1.0 | No additional dependencies |
| Theme sharing | Already solved by JSON (issue #264) | Already solved by JSON |
| Call overhead | Java -> Rust -> ncurses (2 FFI hops) | Java -> ncurses (1 FFI hop) |

The 125 lines of duplicated rendering logic do not justify the introduction of a third programming language, a third build system, platform-specific binary compilation, and an unstable binding generator. The duplication is manageable through documentation and the shared JSON theme schema, which ensures both implementations render the same visual output from the same theme definitions.

## Alternatives Considered

### Rust/UniFFI (This Evaluation)
**Pros:** Single source of truth for rendering logic  
**Cons:** Massive toolchain complexity for 125 lines, 2 MB binary, pre-1.0 binding generator  
**Verdict:** Rejected, cost far exceeds benefit

### Shared C Library (without UniFFI)
**Pros:** Smaller binary (~50 KB), both Java FFM and Python ctypes can call C directly  
**Cons:** Introduces C code maintenance, memory safety risks, similar build complexity  
**Verdict:** Not evaluated in depth; same fundamental problem of disproportionate complexity

### WASM Module
**Pros:** Platform-independent binary, sandboxed execution  
**Cons:** Both Java and Python WASM runtimes are immature, adds another runtime dependency  
**Verdict:** Not viable for terminal rendering that needs direct ncurses access

### Status Quo (JSON Themes + Parallel Implementations)
**Pros:** No new toolchain, standard debugging, zero binary overhead, already working  
**Cons:** ~125 lines of rendering logic duplicated  
**Verdict:** Accepted; the duplication is simple, stable, and well-documented

## Consequences

### Positive Consequences

- **No new toolchain requirements**: Contributors need only Java or Python, not both plus Rust
- **No binary distribution burden**: No platform-specific `.so` files to compile and ship
- **Simpler CI**: No Rust toolchain in CI pipeline
- **Direct debugging**: Standard stack traces and IDE tooling for both implementations
- **Lower latency**: Single FFI hop (Java -> ncurses) instead of double (Java -> Rust -> ncurses)

### Negative Consequences

- **Continued duplication**: ~125 lines of rendering logic remain duplicated between Java and Python
- **Drift risk**: Rendering behavior could diverge if one implementation is updated without the other
- **No single source of truth for rendering**: Theme *definitions* are shared via JSON, but rendering *behavior* is not

### Mitigations for Negative Consequences

1. **Shared test fixtures**: Use the JSON theme files as test fixtures in both projects to verify identical output
2. **Documentation**: Document the rendering algorithm in the theme schema or a shared design document
3. **Visual regression tests**: Screenshot-based tests using the same theme in both implementations to catch drift
